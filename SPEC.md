# Arena3Dweb v3 — Restack Design Spec

**Date:** 2026-04-06
**Status:** Approved for implementation planning

---

## 1. Context & Motivation

Arena3Dweb is a 3D multilayered network visualization tool built with R/Shiny + Three.js. The JavaScript frontend (~4,300 lines of OOP Three.js code) already carries the majority of the application logic. R/Shiny acts as a server that parses files, runs graph algorithms, and bridges the UI.

Key pain points driving the restack:

- `www/js/three/three.js` (51,000 lines) is a manually patched copy of Three.js. The project-specific changes cannot be separated from the library, making upgrades impossible.
- Shiny's reactive/observer model is not designed for building a rich public REST API, which is a stated goal.
- R is a high barrier for contributors unfamiliar with it; the primary maintenance burden falls on one person with occasional master's students.
- Shiny adds latency through its WebSocket reactive model; a stateless REST backend is faster and horizontally scalable.

Primary users are end users on the hosted version (arena3d.org). The deployment stack is invisible to them. The GitHub repository must be preserved — it is cited in two published papers.

---

## 2. Chosen Stack

| Layer | Technology | Replaces |
|---|---|---|
| Frontend build | Vite | Manual script tags in `ui.R` |
| Frontend language | TypeScript | Plain JavaScript |
| 3D engine | Three.js (npm) | Manually patched `three.js` copy |
| UI styling | Bootstrap 5 | Shiny/shinythemes (Darkly) |
| Backend | FastAPI (Python) | R/Shiny |
| Graph algorithms | NetworkX + scipy + python-louvain | igraph (R) |
| API contract | OpenAPI (auto-generated by FastAPI) | None |
| Container | Docker (slim Python base + nginx) | r-base Docker image |

### Why TypeScript over JavaScript

The existing frontend is already OOP (`Scene`, `Layer`, `Node`, `Edge` classes with complex state). TypeScript adds type safety to structure that already exists, provides full Three.js autocomplete via `@types/three`, and makes the codebase self-documenting for contributors. Vite handles TypeScript compilation transparently — no extra tooling overhead.

### Why FastAPI over Node.js

NetworkX mirrors igraph's API closely, making the algorithm port mechanical. Python is the lingua franca in bioinformatics — student contributors almost certainly know it. FastAPI is designed for rich REST APIs with automatic OpenAPI documentation. Node.js graph algorithm libraries are significantly less mature than Python's ecosystem.

## 3. Architecture

The app splits into two independently deployable units communicating over HTTP:

```
┌─────────────────────────────────────────────┐
│  Browser                                    │
│  Vite SPA (TypeScript + Three.js)           │
│  - 3D scene rendering                       │
│  - UI control panels (Bootstrap 5)          │
│  - Typed API client (auto-generated)        │
└────────────────┬────────────────────────────┘
                 │ REST / JSON
┌────────────────▼────────────────────────────┐
│  FastAPI (Python)                           │
│  - Network file parsing & validation        │
│  - Graph algorithms (NetworkX)              │
│  - Session import/export                    │
│  - Public REST API (OpenAPI docs at /docs)  │
└─────────────────────────────────────────────┘
```

**Stateless server.** The FastAPI backend holds zero per-user state between requests. The frontend (TypeScript state store) is the single source of truth. Every algorithm request carries its own complete graph payload; the server is a pure computation service. This means simultaneous users are naturally isolated — no session management, no memory leaks, and the backend is horizontally scalable behind a load balancer without additional infrastructure.

**Serving in production.** nginx serves the static Vite build and proxies `/api/*` to uvicorn. One Docker image, one container. `docker run -p 8080:8080 arena3dweb` works identically to today.

**No heavy UI framework.** The app is a Three.js canvas with control panels. Vanilla TypeScript for panel wiring, Bootstrap 5 for styling. Adding React or Vue on top of Three.js adds complexity without payoff.

**Communication: REST only.** Every interaction is a typed `fetch()` call. FastAPI supports WebSockets natively if streaming progress on large graph computations is needed later — not a day-one requirement.

## 4. Design Patterns

### Frontend

**Command Pattern + CommandHistory** — undo/redo for all scene mutations.

Every user action that mutates scene state becomes a `Command` object capturing how to execute and reverse it. A `CommandHistory` singleton manages the stacks.

```typescript
// src/commands/base.ts
interface Command {
    execute(): void
    undo(): void
    description: string   // shown in UI: "Undo: Apply Force Layout"
}

class CommandHistory {
    private undoStack: Command[] = []
    private redoStack: Command[] = []

    execute(command: Command) {
        command.execute()
        this.undoStack.push(command)
        this.redoStack = []
    }
    undo() {
        const cmd = this.undoStack.pop()
        if (cmd) { cmd.undo(); this.redoStack.push(cmd) }
    }
    redo() {
        const cmd = this.redoStack.pop()
        if (cmd) { cmd.execute(); this.undoStack.push(cmd) }
    }
}
```

Commands to implement: `ApplyLayoutCommand`, `ApplyClusteringCommand`, `ApplyTopologyCommand`, `MoveLayerCommand`, `ChangeNodeColorCommand`, `ChangeNodeSizeCommand`, `ChangeEdgeColorCommand`, `ChangeThemeCommand`, `LoadNetworkCommand`.

File upload and session export are NOT commands — they are initialization/read-only events.

Keyboard shortcuts: `Ctrl+Z` → `history.undo()`, `Ctrl+Shift+Z` → `history.redo()`. Undo/Redo buttons reflect `undoStack.length > 0` / `redoStack.length > 0`.

---

**Event Bus** — decouples UI panels from the Three.js scene.

```typescript
// src/bus/index.ts
bus.emit('node:color-changed', { nodeId: 'A', color: '#ff0000' })
bus.on('node:color-changed', ({ nodeId, color }) => scene.updateNodeColor(nodeId, color))
```

UI panels never import Three.js. The scene never imports UI code. Each side is independently testable. Replaces the role of `rshiny_handlers.js` / `rshiny_update.js`.

---

**State Store (singleton)** — single typed source of truth, replaces scattered Shiny `input$` values and JS globals.

```typescript
// src/store/index.ts
interface AppState {
    config: AppConfig | null
    network: NetworkData | null
    selectedLayers: string[]
    selectedNodes: string[]
    selectedChannels: string[]
    currentTheme: Theme
}
```

UI panels read from the store. Commands update it. The event bus notifies the scene.

### Backend

**Strategy Pattern** — plug-in layout, clustering, and topology algorithms.

```python
# app/services/algorithms/base.py
class LayoutStrategy(ABC):
    @abstractmethod
    def compute(self, graph: nx.Graph, **kwargs) -> dict[str, list[float]]: ...

# app/services/algorithms/registry.py
LAYOUTS: dict[str, LayoutStrategy] = {
    "force":  ForceDirectedLayout(),
    "circle": CircularLayout(),
    "grid":   GridLayout(),
    "random": RandomLayout(),
}
```

Adding a new algorithm = add one file + one line in the registry. The router never changes. Each strategy is independently unit-testable.

**Service Layer** — thin routers, all business logic in services.

```
routers/layout.py     →  validates request, calls services/graph.py
services/graph.py     →  builds nx.Graph, calls algorithm strategy
services/parser.py    →  TSV/JSON parsing and validation
```

## 5. API Contract

### Full API surface

| Method | Endpoint | Replaces |
|---|---|---|
| `GET` | `/api/config` | `handler_initializeGlobals` |
| `POST` | `/api/network` | `handleUploadNetwork()` |
| `POST` | `/api/layout` | `functions/igraph/layout.R` |
| `POST` | `/api/cluster` | `functions/igraph/cluster.R` |
| `POST` | `/api/topology` | `functions/igraph/topology.R` |
| `POST` | `/api/session/import` | `importNetwork()` |
| `POST` | `/api/session/export` | `convertSessionToJSON()` |
| `POST` | `/api/external` | `resolveAPI()` |
| `GET` | `/api/external/<token>` | *(new)* |

OpenAPI docs auto-generated at `/docs`.

### Key data flows

**App initialisation**
```
GET /api/config
← { max_layers, max_edges, max_channels, edge_default_color, channel_colors }
→ Store.update(config)
→ Three.js scene constructed
```

**Network upload**
```
POST /api/network  { file: <multipart TSV> }
← { nodes: [{id, label, layer},...], edges: [{src, trg, weight?, channel?, color?},...],
    layers: [...], warnings: [] }
→ history.execute(new LoadNetworkCommand(response))
```

**Layout request**
```
POST /api/layout  { nodes, edges, algorithm, selected_layers, options }
← { positions: { "NodeA::LayerA": [x, y, z], ... } }
→ history.execute(new ApplyLayoutCommand(response.positions))
```

**Clustering**
```
POST /api/cluster  { nodes, edges, algorithm, selected_layers }
← { clusters: { "NodeA::LayerA": 0, ... }, cluster_count: 4 }
→ history.execute(new ApplyClusteringCommand(response.clusters))
```

**Topology metrics**
```
POST /api/topology  { nodes, edges, metric, selected_layers }
← { values: { "NodeA::LayerA": 0.82, ... }, min: 0.0, max: 1.0 }
→ history.execute(new ApplyTopologyCommand(response.values))
```

**Session export**
```
POST /api/session/export  { layers, nodes, edges, scene }
← application/json download
(no Command — read-only)
```

**Session import**
```
POST /api/session/import  { file: <multipart JSON> }
← { layers, nodes, edges, scene, direction, ... }
→ history.execute(new LoadNetworkCommand(response))
```

**External API**
```
POST /api/external  { nodes, edges, metadata? }
← { url: "https://arena3d.org/?session=<token>" }

GET /api/external/<token>
← full session JSON → same as session import flow
```

The token is a base64-encoded compressed representation of the session JSON — no server-side storage required, keeping the stateless model intact. The URL is self-contained.

### igraph → NetworkX mapping

| R/igraph | Python/NetworkX |
|---|---|
| `graph_from_data_frame()` | `nx.from_pandas_edgelist()` |
| `layout_with_fr()` | `nx.spring_layout()` |
| `layout_with_kk()` | `nx.kamada_kawai_layout()` |
| `layout_in_circle()` | `nx.circular_layout()` |
| `cluster_louvain()` | `community.best_partition()` (python-louvain) |
| `cluster_walktrap()` | `nx.community.asyn_lpa_communities()` (LPA — closest available; exact walktrap port via `igraph` Python bindings if needed) |
| `betweenness()` | `nx.betweenness_centrality()` |
| `transitivity()` | `nx.clustering()` |
| `degree()` | `nx.degree_centrality()` |

## 8. Repository & Migration Strategy

### Branching

```
main      ── R/Shiny version stays live throughout ──────────────────►
               │
               └── tag: v2-legacy
v3        ── new stack developed here ───────────────────────────────►
                                                    │
                                                    └── swap to main when ready
```

Docker Hub continues building from `main` until `v3` is production-ready. The `v2-legacy` tag preserves the R/Shiny version permanently for reproducibility.

### Migration mechanic: port-and-verify

The `v3` branch starts as a copy of `main`. New `frontend/` and `backend/` directories are scaffolded. Old R/Shiny files remain as the **living specification** — read the old file, port it, test it, delete it.

**Deletion = done signal.** At any point, the presence of old files shows exactly what remains to be migrated. `MIGRATION.md` tracks the checklist explicitly.

Example sequence:
```
1. Scaffold backend + frontend (no old files deleted yet)
2. Port + test POST /api/config          → delete config/server_variables.R
3. Port + test POST /api/network         → delete functions/input.R
4. Port + test POST /api/layout          → delete functions/igraph/layout.R
5. Port + test POST /api/cluster         → delete functions/igraph/cluster.R
6. Port + test POST /api/topology        → delete functions/igraph/topology.R
7. Port + test session import/export     → delete functions/init.R, functions/general.R
8. Migrate Three.js classes → TypeScript → delete www/js/classes/
9. Migrate object actions → TypeScript   → delete www/js/object_actions/
10. Migrate event handling + main loop   → delete www/js/event_listeners.js, on_page_load.js
11. Migrate UI panels → index.html + TS  → delete views/, ui.R, server.R, global.R
12. Final cleanup                        → delete www/, functions/, views/, config/ (R)
```

Each step is independently runnable and testable before the next begins.

### Development workflow

```bash
# backend — hot reload
cd backend && uvicorn app.main:app --reload

# frontend — hot reload, /api proxied to backend
cd frontend && npm run dev

# both together
docker-compose up
```
